# Инструкции для ИИ-агентов (AI Development Conventions)

## 👤 Профиль пользователя и проекта
- **Пользователь:** QA Automation Engineer (бэкграунд: Python, Bash, Linux). Общается на профессиональном инженерном языке. Предпочитает глубокие попутные объяснения технологий "слепому" кодингу.
- **Проект:** Защищенная, портируемая инфраструктура 3X-UI + Nginx (Reality + SNI proxying).
- **Парадигма:** Configuration-as-Code, Immutable Infrastructure, Docker-First.

## 🛠 Технологический стек
- **ОС:** Ubuntu 22.04 LTS
- **Core:** 3X-UI v2.9.4+ (Xray core VLESS + Reality TCP)
- **Frontend:** Nginx (модули `stream` для L4 и `http` для L7)
- **Certificates:** Certbot (плагин `certbot-dns-duckdns`, DNS-01 проверка)
- **Firewall:** UFW (взаимодействует с Docker iptables)

## 🚫 Жесткие архитектурные запреты (Guardrails)
1. **No Host Packages:** Все сервисы (Nginx, Certbot, 3X-UI) должны быть строго контейнеризированы. Никакой установки пакетов на хост.
2. **No CLI Updates:** Запрещено использовать внутреннее самообновление 3X-UI через CLI. Обновление — строго через теги образов в Docker-compose.
3. **No Hardcoded Secrets:** Любые токены (DuckDNS) и персональные данные (Email) передаются строго через локальный файл `.env` на хосте. Запрещено коммитить реальные секреты в репозиторий. Все конфигурационные переменные должны строго соответствовать структуре декларативного шаблона `/.env.example` в корне проекта. Скрипты автоматизации обязаны валидировать наличие этих переменных при старте.
4. **No Plain HTTP:** Веб-интерфейсы не должны быть доступны по HTTP без SSL снаружи.
5. **No Human Zone Access (Строгий запрет изоляции):** ИИ-агентам СТРОГО ЗАПРЕЩЕНО открывать, читать, индексировать или использовать файлы из директории `docs/user_docs/`. Эта папка содержит персональные черновики, промпты и заметки PM (Человека). Вся рабочая информация для ИИ находится исключительно в файлах `docs/01_` — `docs/07_`.


## 🔄 Обязательный рабочий процесс (Workflow Rule)

Каждый ИИ-агент при инициализации сессии ОБЯЗАН строго следовать ритуалу синхронизации состояний через Context-as-Code:

1. **Вход в сессию (Handshake):**
   - Первым шагом прочитать `docs/05_interconnection_log.md`.
   - Убедиться, что целевая задача в бэклоге не заблокирована (`Lock`) другим агентом.
   - Обновить блок `## 📌 Текущая активная сессия`, указав свою роль (например, Агент-DevOps), ID задачи из техдолга и текущий статус (`В процессе`).

2. **Выполнение работы:**
   - Реализовать техническую задачу строго в рамках заданных Guardrails.

3. **Выход из сессии (Commit & Handover):**
   - Обновить статус задачи в таблице `docs/06_tech_debt.md`.
   - Обновить дорожную карту в `docs/07_project_status.md`, если этап завершен.
   - Добавить новую запись в `📜 История транзакций (Session Ledger)` в файле `docs/05_interconnection_log.md`. Явно прописать: что сделано, в каком состоянии код и дать конкретные инструкции/контекст для следующего агента (например, для Агента-QA, который будет тестировать фичу).
   - Очистить блок `## 📌 Текущая активная сессия` или перевести его в состояние ожидания.
