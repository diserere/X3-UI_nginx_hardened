# Техническое задание, План исследования и Обучения (Утвержденная версия)

## 1. Исходные данные (Текущее состояние)

* Операционная система: Ubuntu 22.04 LTS на VPS.
* Версия 3X-UI: 2.9.4, установлена непосредственно в хост-систему (не в Docker).
* Текущее распределение портов:
   * Клиентские подключения (Xray): порт 443 (дефолтный).
   * Панель управления (Admin panel): порт 32180.
   * Служба подписок (Subscriptions): порт 32196.
* Доменная инфраструктура: Бесплатный домен 3-го уровня на DuckDNS (xxx.duckdns.org). Поддерживается создание TXT-записей (для DNS-01 проверки Let's Encrypt) и использование произвольных поддоменов 4-го уровня (например, panel.xxx.duckdns.org), которые резолвятся на тот же IP.
* Сетевая защита: Для управления файрволом на сервере используется ufw.
* Профиль пользователя: QA Automation Engineer (Python, Bash, Linux, CLI).

------------------------------
## 2. Цели проекта (Инженерные и Образовательные)

   1. Инженерная: Повысить скрытность и общую защищенность сервиса (панели, подписок и прокси-трафика), спрятав всё за реверс-прокси Nginx и оставив открытым только порт 443.
   2. Архитектурная: Обеспечить максимальную портируемость конфигурации в парадигме Configuration-as-Code.
   3. Образовательная (Технологии): Разобрать глубокую логику работы протоколов (Reality, TCP, TLS handshakes, SNI) и механизмы взаимодействия контейнеров с сетью хоста.
   4. Образовательная (AI-driven SDLC): Практика эффективного управления контекстом больших языковых моделей (LLM), оценка методик декомпозиции задач и синергии человека и ИИ в процессе системного администрирования.

## 3. Направления исследования и Ключевые вопросы

## Пункт 1: Архитектура VLESS + Reality и Nginx SNI-Proxying

* 1.1. Детальный разбор гибридной схемы Nginx (stream + http модули), где Nginx на порту 443 распределяет трафик без расшифровки по SNI: на Reality (внешний сайт-маскировка) и на локальные веб-сервисы. Попутно: как именно Xray и Nginx читают TLS Client Hello до расшифровки трафика.
* 1.2. Оценка безопасности разделения сервисов по поддоменам 4-го уровня DuckDNS (скрытие админки от сканеров).

## Пункт 2: Выпуск Wildcard-сертификатов через DuckDNS

* 2.1. Пошаговый алгоритм автоматического получения и продления Wildcard-сертификата Let's Encrypt (*.xxx.duckdns.org) через Certbot с использованием API DuckDNS для DNS-01 проверки (через TXT-записи). Попутно: почему HTTP-01 проверка здесь не применима.

## Пункт 3: Docker-изация панели 3X-UI и сохранение CLI-функционала

* 3.1. Исследование возможности полноценного использования встроенного скрипта управления x-ui (CLI) внутри Docker-контейнера (через docker exec или аналогичные механизмы).
* 3.2. Сравнительный анализ двух архитектурных подходов:
   * Подход А: Комбинированный (3X-UI на хосте + Nginx в Docker или на хосте).
   * Подход Б: Полный Docker-стэк (3X-UI + Nginx в едином docker-compose.yml с монтированием permanent-директорий для данных и сертификатов).

## Пункт 4: Сетевая безопасность и интеграция с UFW

* 4.1. Решение проблемы обхода правил ufw сетевым стеком Docker (iptables) на низком уровне.
* 4.2. Разработка правил фильтрации для обеих архитектурных схем (Host и Docker), чтобы полностью закрыть порты 32180, 32196 и внутренний порт Reality от внешнего мира, оставив доступ только через Nginx.
