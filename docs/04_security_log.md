# Лог решений по безопасности

## Архитектура маршрутизации трафика (SNI-Proxying)

### Концептуальный вызов
Технология **VLESS + Reality** принципиально несовместима с классическим реверс-проксированием на уровне L7 (HTTP), так как Xray должен контролировать TLS-рукопожатие для эмуляции стороннего сайта-маскировки (например, `yahoo.com`). Прямая терминация TLS на стороне Nginx разрушает логику Reality.

### Решение: Маршрутизация на уровне L4 (TCP)
Для одновременной работы Reality, панели управления и службы подписок на одном внешнем порту (`443`) применяется модуль Nginx `stream` с директивой `ssl_preread on`. 

Nginx считывает поле **SNI (Server Name Indication)** из пакета `Client Hello` на транспортном уровне до того, как произойдет расшифровка трафика, и перенаправляет сырой TCP-поток на соответствующий бэкенд.

#### Схема распределения потоков данных

```text
                     ┌────────────────────────────────────────┐
                     │ Входящий трафик из WAN (Порт 443, TCP) │
                     └───────────────────┬────────────────────0;
                                         │
                                         ▼
                     ┌────────────────────────────────────────┐
                     │           Nginx Stream (L4)            │
                     │            (ssl_preread on)            │
                     └───────────────────┬────────────────────0;
                                         │
                 ┌───────────────────────┴───────────────────────┐
                 │ Разделение по имени домена (SNI)              │
                 ▼                                               ▼
     [ SNI: panel.xxx.duckdns.org ]                  [ SNI: yahoo.com ]
     [ SNI: sub.xxx.duckdns.org   ]                  (или другой сайт-маскировка)
                 │                                               │
                 ▼                                               ▼
┌─────────────────────────────────┐             ┌─────────────────────────────────┐
│        Nginx HTTP (L7)          │             │          Xray Backend           │
│  (Внутренний порт, например 443)│             │     (Внутренний порт: 8443)     │
└────────────────┬────────────────┘             └─────────────────────────────────┘
                 │
         ┌───────┴───────┐
         ▼               ▼
   [Путь: /admin/]  [Путь: /sub/]
         │               │
         ▼               ▼
   ┌───────────┐   ┌───────────┐
   │ Панель    │   │ Служба    │
   │ (32180)   │   │ Подписок  │
   │           │   │ (32196)   │
   └───────────┘   └───────────┘
```

### Преимущества для безопасности
1. **Минимизация Attack Surface:** Снаружи открыт строго один порт — `443`. Порты панели и подписок полностью изолированы.
2. **Защита от сканирования по IP:** При попытке прямого обращения к IP-адресу сервера (без SNI) срабатывает правило `default`, которое отдает пустую заглушку, скрывая присутствие VPN-сервисов.
3. **Изоляция Reality:** VPN-трафик проходит через Nginx транзитом без дешифрации, что исключает появление дополнительных векторов атак на стороне веб-сервера.

## Обеспечение сетевой безопасности на уровне ядра (Docker iptables & UFW bypass mitigation)

### Контекст
Сетевой стек Docker при пробросе портов наружу (`ports: - "X:Y"`) модифицирует цепочки `iptables` на системном уровне в обход правил высокоуровневого файрвола UFW. Это создает уязвимость, при которой порты бэкенда могут оказаться открытыми всему миру, вопреки запрещающим политикам `ufw deny`.

### Принятое решение (Security by Design)
Проблема несанкционированного доступа к компонентам бэкенда решена на этапе проектирования архитектуры Docker-сетей:

1. **Изоляция портов бэкенда:** Для сервиса `3x-ui` в `docker-compose.yml` полностью исключена директива `ports`. Внутренние порты панели (`32180`, `32196`) и порт Reality (`8443`) не публикуются в хост-системе. Следовательно, Docker не создает для них разрешающих правил в цепочке `FORWARD` брандмауэра.
2. **Публикация единственной точки входа:** Наружу через Docker публикуется исключительно стандартный порт веб-трафика `443:443` для контейнера `nginx`. Данный порт обязан быть открытым для обработки легитимных подключений.
3. **Конфигурация UFW на хосте:** На самой VPS брандмауэр UFW настраивается в режиме максимального ограничения, защищая только базовые системные службы хоста:
   ```bash
   sudo ufw default deny incoming
   sudo ufw default allow outgoing
   sudo ufw allow 22/tcp comment 'SSH access'
   sudo ufw allow 443/tcp comment 'Nginx Docker Frontend'
   sudo ufw enable
   ```
4. **Вывод старой инсталляции:** Предыдущая хост-версия панели `3X-UI` полностью демонтируется из системы (`systemctl disable x-ui`), чтобы исключить конфликты портов и потенциальные скрытые векторы атак на уровне системных демонов.
