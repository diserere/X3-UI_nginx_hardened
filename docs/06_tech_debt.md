# Технический долг и Беклог задач (Tech Debt & Backlog)

## 📌 Критические инженерные риски (Выявлено в ходе архитектурного ревью)

- [ ] **[Type: Bug] [Scope: Certbot/Nginx] [Priority: High]** Автоматический reload конфигурации Nginx при обновлении SSL.
  - *Описание:* Nginx кэширует сертификаты в RAM. Когда Certbot обновит файлы в `./data/certs/`, Nginx продолжит использовать старые до ручного перезапуска.
  - *Решение:* Доработать `src/certbot/entrypoint.sh` или настроить cron внутри контейнера Nginx для выполнения `nginx -s reload` раз в сутки.

- [ ] **[Type: Security] [Scope: 3X-UI] [Priority: High]** Конфигурация локальных секретных URI в панели.
  - *Описание:* Для защиты от сканирования необходимо изменить дефолтные пути `/` в настройках 3X-UI на сложные рандомные хэши (например, `/admin/secret-qaz-123/` и `/sub/secret-ws-456/`).
  - *Решение:* Сделать это вручную через веб-интерфейс при первом запуске и зафиксировать схему путей в закрытых заметках.

- [ ] **[Type: Feature] [Scope: Host/Security] [Priority: Medium]** Интеграция Fail2Ban для защиты от брутфорса админки.
  - *Описание:* Защита панели от перебора паролей. Поскольку Nginx работает в L7, он пишет реальные IP атакующих в лог.
  - *Решение:* Смонтировать логи Nginx на хост `./data/logs/nginx/access.log` и настроить Fail2Ban на хост-системе для парсинга ошибок `401/404` и блокировки IP через `iptables`.

- [ ] **[Type: DevOps] [Scope: Backup] [Priority: Medium]** Автоматизация бэкапов базы данных `x-ui.db`.
  - *Описание:* База данных пользователей и настроек является критической точкой отказа (Single Point of Failure).
  - *Решение:* Написать Bash/Python скрипт на хосте, который по cron делает дамп `./data/3x-ui/x-ui.db`, архивирует его и отправляет в безопасное место (приватный репо / Telegram).
