# Лог архитектурных решений (ADR)

## Решение №1: Переход на полный Docker-стэк и отказ от встроенного CLI 3X-UI

### Контекст
Встроенный CLI-скрипт управления панели 3X-UI (`x-ui`) жестко завязан на управление системными службами через `systemctl` (Systemd). Внутри Docker-контейнера Systemd отсутствует, из-за чего скрипт возвращает некорректный статус `Not installed` для работающей панели.

### Принятое решение
Принято решение отказаться от использования встроенного CLI в пользу концепции **Immutable Infrastructure (Неизменяемая инфраструктура)**. Вся система разворачивается строго через Docker-compose. Функционал CLI полностью компенсируется нативными инструментами контейнеризации и операциями на хост-системе.

### Шпаргалка по замене функций CLI в Docker-окружении

1. **Мониторинг и логирование:**
   Вместо `x-ui status` / `x-ui log`:
   ```bash
   docker compose ps
   docker compose logs -f 3x-ui
   ```

2. **Экстренный сброс учетных данных администратора:**
   Вместо `x-ui` (пункт сброса пароля). Так как база данных SQLite монтируется как Volume на хост, сброс выполняется прямой SQL-инъекцией в локальный файл БД:
   ```bash
   sqlite3 ./data/x-ui.db "UPDATE users SET username='admin', password='admin' WHERE id=1;"
   ```

3. **Изменение портов панели «вслепую»:**
   Вместо изменения портов через CLI, конфигурация портов переносится в декларативный вид в файле `docker-compose.yml` в секцию `ports`. При необходимости порт меняется там, после чего выполняется:
   ```bash
   docker compose up -d --force-recreate 3x-ui
   ```

4. **Обновление панели:**
   Функция самообновления внутри CLI является антипаттерном Docker, так как изменения стираются при перезапуске контейнера. Обновление версии панели теперь контролируется исключительно сменой тега Docker-образа в `docker-compose.yml`.

## Решение №2: Контейнеризация Certbot и DNS-01 проверка через DuckDNS API

### Контекст
Для работы реверс-прокси по схеме разделения SNI требуется SSL-сертификат для поддоменов 4-го уровня. Классическая HTTP-01 проверка требует открытия порта 80, что деанонимизирует сервер перед сканерами ТСПУ и нарушает скрытность инфраструктуры.

### Принятое решение
Выпуск сертификатов полностью перенесен в изолированный контейнер Certbot с плагином `certbot-dns-duckdns`. 
1. **Метод проверки:** Выбран DNS-01 (через API DuckDNS). Проверка подлинности владения доменом происходит через динамическое создание TXT-записи в DNS-зоне. Порты на сервере для проверки не открываются.
2. **Тип сертификата:** Выпускается один Wildcard-сертификат (`*.xxx.duckdns.org`), закрывающий любые потребности в поддоменах для админки и подписок.
3. **Автоматизация (Configuration-as-Code):** Логика работы вынесена из однострочника Docker-compose в чистый Bash-скрипт `src/certbot/entrypoint.sh`, который монтируется внутрь контейнера. Скрипт раз в 12 часов проверяет необходимость обновления.
4. **Безопасность учетных данных:** Токен DuckDNS и Email передаются через системный файл `.env` (скрыт из репозитория). Скрипт `entrypoint.sh` на лету генерирует временный `.ini` файл внутри контейнера с правами `600`, исключая компрометацию токена через Docker-инспекцию.
5. **Хранилище:** Сертификаты сохраняются через Bind Mount в директорию `./data/certs` на хосте, обеспечивая прозрачность для бэкапов и легкий доступ для Nginx в режиме Read-Only.

## Решение №3: Отказ от L4-Only схемы в пользу гибридной L4/L7 для защиты от TLS Active Probing

### Контекст
В процессе оптимизации рассматривался вариант полной ликвидации L7-модуля Nginx (HTTP) и маршрутизации веб-сервисов (админки и подписок) напрямую через L4 (`stream`) на порты 3X-UI. Панель 3X-UI технически способна сама терминировать SSL и фильтровать входящие URI.

### Обнаруженная уязвимость (TLS Active Probing & Сертификатная корреляция)
В схеме L4-Only при активном сканировании (Active Probing) со стороны ТСПУ/цензоров на порту `443` возникает критическая аномалия:
1. При запросе легитимного SNI (`panel.xxx.duckdns.org`) сервер завершает TLS-рукопожатие и выдает сертификат Let's Encrypt для DuckDNS.
2. При запросе SNI сайта-маскировки Reality (`yahoo.com`) на том же порту `443` сервер перенаправляет поток в Xray, который выдает оригинальный сертификат `yahoo.com`.
Наличие двух принципиально разных SSL-сертификатов на одном IP/порту является 100% маркером VPN-сервера Reality и приводит к мгновенной блокировке со стороны ТСПУ.

Дополнительно на уровне L4 невозможно модифицировать HTTP-заголовки, что лишает панель 3X-UI информации о реальных IP-адресах клиентов (`X-Real-IP`), делая невозможным аудит безопасности и автоматическую блокировку брутфорсеров (Fail2Ban).

### Принятое компромиссное решение
1. **Сохранить гибридную схему L4/L7.** Модуль L7 (Nginx HTTP) выступает в роли «щита», изолирующего сертификат DuckDNS от Reality-трафика.
2. **Устранить дублирование путей.** В конфигурации Nginx разделение админки и подписок происходит строго по доменам 4-го уровня (`panel.` и `sub.`), без парсинга URI-путей на стороне веб-сервера.
3. **Делегирование путей:** Секретные URI-пути настраиваются исключительно внутри 3X-UI для обеспечения локальной безопасности приложения. Nginx прозрачно транслирует их бэкенду.

## Решение №4: Монолитная декларативная инфраструктура (Configuration-as-Code) через Docker Compose

### Контекст
Для обеспечения стопроцентной портируемости системы и изоляции компонентов друг от друга требовалось объединить подсистему генерации сертификатов (Certbot), фронтенд (Nginx) и бэкенд (3X-UI) в единый управляемый контур, исключая ручную установку зависимостей в хост-систему VPS.

### Принятое решение
Вся инфраструктура описана декларативно в файле `docker-compose.yml` в корне репозитория.

1. **Сетевая изоляция (Docker Network Bridge):** Создана кастомная изолированная сеть `vpn_network`. Контейнеры общаются между собой по внутренним именам сервисов через встроенный Docker DNS.
2. **Защита бэкенда на уровне контейнеризации:** В секции сервиса `3x-ui` полностью отсутствует директива `ports`. Внутренние порты панели (`32180`, `32196`) и порт Reality (`8443`) **не пробрасываются наружу**. Они физически недоступны из внешней сети WAN хоста. Единственной точкой входа в систему является порт `443` контейнера Nginx.
3. **Обмен данными через Bind Mounts:** Папка `./data/certs` на хосте используется как общий Read-Only буфер между Certbot и Nginx. Папка `./src/nginx/html` монтируется в Nginx для отдачи статической заглушки ("Under Construction") при обращениях к корню домена без указания секретных URI.
